# 分布式异步记忆分类系统设计方案 (Distributed Async Memory System)

## 1. 架构概述
本方案采用“边缘采集 + 远程推理”的架构，解决低配边缘设备在处理复杂 LLM 分类任务时产生的响应延迟和资源不足问题。

### 核心组件
*   **采集端 (Edge Node)**: 运行 OpenClaw Gateway 和 `memory-lancedb-pro` 插件。负责消息拦截与初步过滤。
*   **推理端 (Inference Provider)**: 运行 Ollama 服务，提供轻量级大模型（如 `qwen3.5:2b`）推理能力。
*   **通信协议**: 基于标准 HTTP 的 Ollama API 接口。

## 2. 关键特性

### A. 异步非阻塞流程 (Async Fire-and-Forget)
为了不影响用户的对话实时性，插件的 `agent_end` 钩子采用异步模式：
1.  Agent 生成响应后，立即触发 `agent_end` 钩子。
2.  插件在后台开启一个脱离主流程的 Promise 任务。
3.  插件立刻向 Gateway 返回，确保用户能第一时间收到回复。
4.  分类、向量化和入库操作在后台静默完成。

### B. 推理负载卸载 (Inference Offloading)
*   **资源优化**: 将计算密集型的 LLM 分类任务从低配节点卸载到拥有更强计算资源（如更多 CPU 核心或 GPU）的推理节点。
*   **配置方式**: 插件 `baseURL` 指向内网推理服务器地址。

### C. 智能过滤与去重
*   **噪音过滤**: 通过关键词过滤机制拦截无效或测试信息。
*   **语义去重**: 入库前利用向量数据库（如 LanceDB）进行相似度检索，防止重复存储冗余对话块。

## 3. 部署与维护规范

### 插件修改生效流程
由于 TypeScript 编译缓存机制，修改插件代码后建议执行以下步骤：
1.  清除编译缓存：`rm -rf ~/.openclaw/.jiti`
2.  重启 Gateway 服务。

### 模型选择建议
*   **推荐模型**: `qwen3.5:2b`，在分类准确率与推理速度之间具有较好的平衡。
*   **空间管理**: 定期清理模型层缓存文件，保持系统盘空间。

## 4. 后续演进
1.  **协议增强**: 接入 `openprose` 和 `lobster` 协议，增强多智能体协同能力。
2.  **存储合并**: 支持跨节点记忆导出与合并，构建统一的知识库。
